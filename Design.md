# 教研论文智能体系统 Design

## 1. 文档目标

这不是教程，也不是逐步抄代码手册。

这份文档只回答四件事：

1. 这个项目到底做什么
2. 最小可用版本做到什么程度
3. 系统边界和技术选型是什么
4. 团队应该按什么结构实现，才不容易返工

---

## 2. 项目定义

本项目是一个“固定三节点”的教研论文工作流系统。

用户在前端通过可视化方式配置一条固定链路：

`文献输入 -> 主题分析 -> 提纲生成`

点击运行后，前端将当前工作流数据发送到后端；后端按既定顺序执行分析逻辑，返回：

- `summary`
- `keywords`
- `outline`

前端负责展示工作流和结果，后端负责校验、编排和模型调用。

---

## 3. MVP 范围

### 3.1 本期必须完成

最小可用版必须包含以下能力：

1. 前端可看到左中右三栏布局
2. 左侧可拖入 3 种节点
3. 中间画布可放置节点、建立连线
4. 只允许合法连线：
   `文献输入 -> 主题分析 -> 提纲生成`
5. 点击节点后，右侧可编辑该节点参数
6. 点击“运行测试”后，前端调用后端接口
7. 后端校验工作流结构是否合法
8. 后端顺序执行三步逻辑并返回结果
9. 前端展示摘要、关键词、提纲
10. 前端能显示基础执行状态：
   `idle / running / success / error`

### 3.2 本期明确不做

以下内容不属于 MVP，避免项目失控：

1. 任意拓扑工作流
2. 动态增加节点类型
3. 多分支、多回路、多输入输出
4. 数据库持久化
5. 用户系统
6. 工作流保存/分享/版本管理
7. 真正的文献检索能力
8. 文件上传解析 PDF/Word
9. 流式输出
10. 复杂权限和部署体系

一句话说清楚：这期不是“通用智能体平台”，只是“固定三步的可视化论文辅助流程”。

---

## 4. 为什么这样收缩

这个项目天然只有三步、单链路、固定产物。

如果一开始就按“通用工作流平台”思路设计，复杂度会远大于业务价值，最后最容易发生三件事：

1. 前端做成好看的假界面，但没有真实运行能力
2. 后端堆了 LangGraph 和模型调用，但前端接不起来
3. 团队花很多时间在架构词汇上，真实交付非常慢

所以本设计坚持两个原则：

1. 先把固定流程跑通，再谈抽象
2. 先定义契约，再写页面和代码

---

## 5. 系统边界

### 5.1 前端职责

前端只负责：

1. 节点拖拽与连线交互
2. 节点参数编辑
3. 工作流结构校验的第一层拦截
4. 发起运行请求
5. 显示执行状态和结果

前端不负责：

1. 调用大模型
2. 编排执行逻辑
3. 决定业务真相

### 5.2 后端职责

后端只负责：

1. 接收前端工作流 JSON
2. 校验结构是否合法
3. 构造执行状态
4. 顺序执行分析步骤
5. 调用模型或降级规则逻辑
6. 返回统一结果

后端不负责：

1. 管理前端 UI 状态
2. 存储页面布局
3. 处理复杂账户体系

---

## 6. 技术选型

### 6.1 前端

- `React`
- `TypeScript`
- `@xyflow/react`

选择理由：

1. 画布交互、节点、连线是核心需求
2. React Flow 足够覆盖本项目，不需要自造画布系统

### 6.2 后端

- `FastAPI`
- `Pydantic`

选择理由：

1. 快速定义接口
2. 请求体校验清晰
3. 本项目接口数量极少，不需要更重框架

### 6.3 流程编排

- `LangGraph`

选择前提：

当前项目只有固定三步，理论上不用 LangGraph 也能完成。

但如果目标是后续继续扩展“节点式智能体流程”，那引入 LangGraph 是合理的，只是必须控制它的使用范围：

1. LangGraph 只负责后端执行顺序
2. 不要把它当作前端工作流 JSON 的直接镜像
3. 不要为了“图”而再造一套动态 DSL

也就是说，前端的图是编辑体验，后端的图是执行实现，两者相关，但不是一回事。

### 6.4 模型层

- 首选通过统一函数封装模型调用
- 没有可用密钥时必须能降级运行

原因：

如果没有 fallback，团队会把大量时间浪费在“为什么模型没响应”这种非核心问题上。

---

## 7. 页面结构

MVP 统一采用单页工作台布局。

### 7.1 页面区域

1. 顶部操作栏
   包含运行按钮、重置按钮
2. 左侧节点库
   展示 3 个可拖拽节点
3. 中间画布
   放置节点、连接节点
4. 右侧参数面板
   编辑当前选中节点参数
5. 底部结果面板
   展示 `summary / keywords / outline`

### 7.2 不建议的页面形态

以下形态会明显偏离目标：

1. 多标签页管理“工作流 / 编辑器 / 资源中心”
2. 把工作流页做成静态展示海报
3. 只有高保真视觉，没有真实节点状态

因为本项目核心不是“学术平台感”，而是“工作流是否真正能跑”。

---

## 8. 固定节点定义

### 8.1 文献输入节点

用途：

收集研究主题和原始文献文本。

参数：

- `topic: string`
- `documents: string`

输出：

- `topic`
- `documents`

### 8.2 主题分析节点

用途：

根据主题和文献文本生成摘要与关键词。

参数：

- `keywordCount: number`
- `prompt: string`

输出：

- `summary: string`
- `keywords: string[]`

### 8.3 提纲生成节点

用途：

根据摘要和关键词生成论文提纲。

参数：

- `level: number`
- `paperType: string`
- `prompt: string`

输出：

- `outline: string`

## 9. 数据契约

这是整份设计里最重要的部分。

如果这里不稳定，前后端一定返工。

### 9.1 前端发送给后端的请求体

```json
{
  "nodes": [
    {
      "id": "input-1",
      "type": "input",
      "position": { "x": 120, "y": 180 },
      "data": {
        "label": "文献输入",
        "status": "idle",
        "config": {
          "topic": "人工智能在小学教育中的应用",
          "documents": "..."
        }
      }
    }
  ],
  "edges": [
    {
      "id": "edge-1",
      "source": "input-1",
      "target": "analysis-1"
    }
  ]
}
```

### 9.2 后端返回给前端的成功结构

```json
{
  "summary": "……",
  "keywords": ["人工智能", "小学教育"],
  "outline": "一、研究背景\n二、研究方法"
}
```

### 9.3 后端错误返回结构

使用标准 HTTP 错误码，错误体至少包含：

```json
{
  "detail": "请按“文献输入 -> 主题分析 -> 提纲生成”的顺序连线。"
}
```

### 9.4 统一约束

必须统一以下命名：

- 前端节点类型：`input | analysis | outline`
- 后端节点类型：`input | analysis | outline`
- 不允许一会儿叫 `inputNode`，一会儿叫 `input`

这是低级但高频的返工来源。

## 10. 前端架构

### 10.1 推荐文件结构

```text
src/
├── App.tsx
├── main.tsx
├── components/
│   ├── FlowHeader.tsx
│   ├── Sidebar.tsx
│   ├── FlowCanvas.tsx
│   ├── ParameterPanel.tsx
│   ├── ResultPanel.tsx
│   ├── WorkflowContext.tsx
│   └── nodes/
│       ├── InputNode.tsx
│       ├── AnalysisNode.tsx
│       └── OutlineNode.tsx
├── data/
│   └── nodeTemplates.ts
├── types/
│   └── workflow.ts
└── utils/
    ├── validateWorkflow.ts
    └── runWorkflow.ts
```

### 10.2 前端状态原则

前端必须有一个统一状态源，至少包含：

- `nodes`
- `edges`
- `selectedNodeId`
- `result`
- `error`
- `isRunning`

不要把同一份业务状态散落在多个组件的本地 `useState` 里。

否则最先出问题的就是：

1. 参数面板改了，节点卡片没更新
2. 节点卡片改了，运行请求还是旧数据
3. 运行完成了，结果面板和节点状态不同步

### 10.3 前端校验分两层

第一层：交互校验

- 非法连线直接不允许建立

第二层：运行前校验

- 即使界面上能操作，也必须在点击运行前再验证一次

因为用户可能会删边、删节点、留下空参数。

---

## 11. 后端架构

### 11.1 推荐文件结构

```text
backend/
├── main.py
├── workflow.py
├── requirements.txt
└── .env
```

### 11.2 main.py 负责什么

`main.py` 只做四件事：

1. 注册 FastAPI 和 CORS
2. 定义请求模型
3. 校验请求合法性
4. 调用 `workflow.py` 并返回结果

不要把大段模型提示词和业务逻辑全部堆进 `main.py`。

### 11.3 workflow.py 负责什么

`workflow.py` 只做三件事：

1. 定义执行状态 `WorkflowState`
2. 定义三个节点函数
3. 构建并编译执行图

### 11.4 是否必须使用 LangGraph

不是必须。

如果团队只是想尽快交付 MVP，可以先用普通 Python 函数组合实现：

1. `read_input`
2. `analyze`
3. `outline`

当这三步稳定后，再替换成 LangGraph。

如果从 Day 1 就上 LangGraph，也可以，但要承认一个事实：

这不是为了 MVP 必需，而是为了后续扩展预留。

## 12. 执行流程

一次运行的标准流程如下：

1. 用户在前端放置 3 个节点
2. 用户建立合法连线
3. 用户填写参数
4. 用户点击“运行测试”
5. 前端先执行本地校验
6. 前端发送 `{nodes, edges}` 到 `/api/run-workflow`
7. 后端校验结构和参数
8. 后端构造初始状态
9. 后端顺序执行：
   `read_input -> analyze -> outline`
10. 后端返回结果
11. 前端更新结果面板和节点状态

---

## 13. 执行状态设计

每个节点统一使用以下状态值：

- `idle`
- `running`
- `success`
- `error`

### 13.1 MVP 阶段的简化策略

MVP 可以先这样做：

1. 点击运行后，所有节点先进入 `running`
2. 请求成功后，所有节点变成 `success`
3. 请求失败后，所有节点变成 `error`

### 13.2 下一阶段的增强

后续再做逐节点状态流转：

1. 输入节点完成后变 `success`
2. 分析节点进入 `running`
3. 生成节点最后完成

不要在第一版就追求完整执行可视化，否则实现成本和调试复杂度会明显增加。

---

## 14. 错误处理

必须覆盖以下错误场景：

1. 缺少节点
2. 连线不合法
3. 输入文本为空
4. 后端未启动
5. 模型调用失败
6. 后端内部异常

处理原则：

1. 前端给出可理解的中文提示
2. 后端返回标准错误码
3. 模型失败时优先回退到规则化结果，而不是直接让整条链路崩掉

---

## 15. 测试要求

### 15.1 前端至少验证

1. 三种节点可以拖入画布
2. 非法连线被拦截
3. 点击节点后参数面板显示正确
4. 修改参数后节点数据被正确更新
5. 点击运行后能看到结果或错误提示

### 15.2 后端至少验证

1. `/ping` 返回正常
2. `/api/run-workflow` 能接收合法请求
3. 缺失节点时返回 400
4. 连线错误时返回 400
5. 合法请求时返回 `summary / keywords / outline`

---

## 16. 开发顺序

推荐顺序如下：

### Phase 1：先打通前端基础交互

1. 空白画布
2. 节点库
3. 拖拽放置
4. 自定义节点
5. 参数面板
6. 连线校验

### Phase 2：打通前端本地运行

1. 本地校验函数
2. 假数据结果面板
3. 基础状态显示

### Phase 3：接后端

1. FastAPI 空壳
2. `/api/run-workflow`
3. 后端校验
4. 先用普通函数跑通
5. 再接 LangGraph

### Phase 4：接模型

1. 封装模型调用
2. 加入 fallback
3. 调整提示词

关键原则：

不要在前后端都没跑通时就急着接真实模型。

---

## 17. 交付标准

项目只有在同时满足以下条件时，才算完成 MVP：

1. 页面不是静态海报，而是真正可操作的工作流
2. 运行按钮不是假按钮，而是能触发真实请求
3. 后端不是只返回硬编码字符串，而是按状态执行完整流程
4. 没有模型密钥时系统也能给出可演示结果
5. README 能让其他人独立启动前后端

---

## 18. 最终判断

这个项目的本质，不是“做一个酷炫的智能体平台”。

它的本质是：

在尽量小的复杂度下，交付一条真正能运行的“可视化论文辅助流程”。

如果实现过程中出现以下倾向，就说明项目走偏了：

1. UI 很复杂，但不能真正运行
2. 架构名词很多，但接口契约不稳定
3. 过早抽象成通用平台
4. 团队在修框架问题，而不是在交付业务结果

正确路线只有一条：

先把固定三节点流程做扎实，再谈抽象和扩展。
